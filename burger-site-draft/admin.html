Account not authorized

Your account isn't an admin. Sign in with a chef account.

=========================================================================== Prerequisites: • A chef account exists in Supabase Authentication. • That account has app_metadata.role = 'admin' (set via Supabase Studio → Authentication → Users → raw_app_meta_data or via SQL: UPDATE auth.users SET raw_app_meta_data = raw_app_meta_data || '{"role":"admin"}'::jsonb WHERE email = 'chef@...') • supabase-config.js is filled in (URL + publishable key). • The three SQL migrations (001/002/003) have been run in Supabase. Steps (run in Chrome DevTools unless noted): --------------------------------------------------------------------------- 1. Clear all site data for the admin.html origin (Application tab → Storage → Clear site data) to ensure no residual session. Open admin.html — login form should be visible; shell and role-mismatch sections must be hidden. 2. Enter a valid chef email in the login field and submit. In Network tab, confirm a request to /auth/v1/otp returns HTTP 200. The status element must show "Check your inbox for the login link." within the login section (not in shell or mismatch sections). 3. Open the magic link from the inbox (or use Supabase Studio → Authentication → Users → "Send magic link" as fallback). The link loads admin.html with an access_token in the URL fragment; detectSessionInUrl: true parses it and establishes the session. Admin shell should appear immediately; topbar shows the chef's email and the "Sign out" button is visible. 4. Reload admin.html without clearing storage. The shell must remain visible (session restored from localStorage). 5. Sign out via the topbar "Sign out" button. Login form reappears; shell and role-mismatch are hidden. 6. Pre-condition for next two tests: sign in with a NON-admin account (app_metadata.role is absent or is not 'admin'). Role-mismatch section appears with "Account not authorized" message and a "Sign out" button. The shell is hidden. 7. From the role-mismatch view, click "Sign out". Return to the login form — no shell or mismatch visible. 8. (Optional / DevTools only) Manually inject an admin session token via browser console: localStorage.setItem('sb--auth-token', JSON.stringify({ token: { access_token: '…', provider_token: null, user: { id: '…', email: 'admin@…', app_metadata: { role: 'admin' } } } })) Refresh — shell should appear immediately without the login flow. 9. RLS perimeter test (requires two browser profiles or incognito): — Incognito (no session): open admin.html. In DevTools console run: supabase.from('orders').select('*') Expected: HTTP 200 with zero rows (anon INSERT policy lets you INSERT but RLS blocks reads until signed in). — Admin session: same console call returns all order rows visible to the admin policy. 10. No secret-key strings in browser-loaded files: In burger-site-draft/, search for the Supabase secret-key prefix and for JWT-like Bearer token strings. Both searches must return zero matches (exit 1). =========================================================================== -->
